# Bike Demand Prediction Dashboard (Flask + ML)

Production-ready MVP dashboard to upload bike-sharing datasets, preprocess data, train regression models, evaluate performance, visualize analytics, and make predictions.

## Features
- Upload CSV/Excel datasets and preview/summary
- Preprocess (missing values, one-hot encoding, scaling, train/test split)
- Train models: Linear Regression, Decision Tree, Random Forest, Gradient Boosting (+ XGBoost if installed)
- Metrics: R², Adjusted R², MAE, MSE, RMSE
- 10+ charts via Chart.js (and small plugins for heatmap/boxplot)
- Prediction form generated dynamically from model features
- SQLite persistence: datasets, trained models, prediction history
- Export predictions/models as CSV

## Quickstart (local)
```bash
cd bike-demand-dashboard
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
python app.py
```

Open `http://127.0.0.1:5000/dashboard`.

Recommended steps:
```bash
cd bike-demand-dashboard

# 1. make sure your code is committed
git status

# 2. authenticate with Andasy CLI
andasy auth login

# 3. create the app if it does not exist yet
andasy apps create machine-learning

# 4. set secrets / env vars
andasy secrets set SECRET_KEY="change-this-secret"
# optional
andasy secrets set GEMINI_API_KEY="your-key"

# 5. deploy
andasy deploy
```

After deploy, open the app URL from the Andasy CLI output.

Notes:
- The container listens on port `8080`.
- Production uses `gunicorn`.
- The default SQLite database is file-based. If Andasy restarts the container without persistent storage, uploaded files, trained models, and DB data can be lost.
- For durable production data, switch `DATABASE_URL`, uploads, and model storage to persistent storage or managed services.

## Deploy on Render
This repo now includes a root-level [`render.yaml`](/home/zera/Documents/Final%20Ml%20pro/render.yaml) blueprint that deploys the Flask app from `bike-demand-dashboard/`.

Recommended steps:
```bash
# from the repository root
git status

# push your latest changes to GitHub, then in Render:
# 1. New + -> Blueprint
# 2. Select this repository
# 3. Render will detect render.yaml
# 4. Review the generated web service and create it
```

What the blueprint does:
- Uses `bike-demand-dashboard/` as the Render service root
- Installs dependencies from `requirements.prod.txt`
- Starts the app with `gunicorn` using `wsgi:app`
- Exposes health checks at `/api/health`
- Mounts a persistent disk at `/var/data`
- Stores SQLite DB, uploads, exports, and trained models on that persistent disk

Important environment notes:
- `SECRET_KEY` is auto-generated by Render from the blueprint
- `GEMINI_API_KEY` is optional and should be added manually in the Render dashboard if you want AI summaries
- If you later switch to Render Postgres, set `DATABASE_URL` in the service environment and it will override SQLite automatically

Persistent storage notes:
- The blueprint stores app data at `/var/data` so model files and uploaded datasets survive redeploys/restarts
- The included blueprint uses the `starter` plan because Render persistent disks are for paid web services, not Free web services
- If you want the Free plan instead, remove the `disk` block and avoid relying on local SQLite/uploads for durable data

## Example dataset format
Any bike-demand style dataset works as long as:
- It’s a tabular CSV/Excel file
- The **target column** is numeric (e.g., `count`, `demand`, `rides`)

Recommended columns (common in bike sharing):
- `datetime` (or `date`, `timestamp`)
- `temp`, `humidity`, `windspeed`
- `season`, `weather`, `holiday`, `workingday`
- Target: `count` (rides/demand)

An example CSV is included at `example_data/bike_demand_example.csv`.
